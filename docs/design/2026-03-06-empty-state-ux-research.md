# 빈 상태(Empty State) UX 리서치 & 권장안

> **작성**: 2026-03-06  
> **목적**: 분석 완료 → 추천 0건/로드 실패 시 초기 유저 이탈 방지. 목데이터 없이 다른 방법으로 빈 상태를 설계.  
> **관점**: Product Designer + PM 리서치 협업

---

## 1. 현재 상황 정리

### 1.1 화면별 원인

| 화면 | 메시지 | 추정 원인 |
|------|--------|-----------|
| **분석 완료 → 추천 리스트** | "추천 프로필을 불러오지 못했어요" (에러) | `generate-daily-recommendations` Edge Function 미배포/실패 → `ensureDailyRecommendations(isInitial: true)` 실패 → AsyncError |
| **분석 완료 → 추천 리스트** | "아직 매칭 상대가 없어요" (빈 리스트) | `saju_compatibility`에 데이터 없음(배치 미실행/이성 유저 0) 또는 `daily_matches` 0건 → data는 오지만 빈 리스트 |
| **홈 "받은 좋아요"** | "아직 없어요" | 실제로 받은 좋아요 0건 (정상 빈 상태) |

### 1.2 궁합 점수가 안 나오는 이유 (가설 검증)

- **맞음.** DB에 궁합을 계산할 수 있는 값이 없거나, 파이프라인이 아직 안 돌았을 가능성이 큼.
  - `batch-calculate-compatibility`: 방금 배포됨. 사주 분석 직후 비동기로 돌므로 **동시에** 추천을 보려면 이미 계산된 행이 있어야 함.
  - `generate-daily-recommendations`: **아직 프로젝트에 배포되지 않음** → 호출 시 404 → "추천 프로필을 불러오지 못했어요"로 이어짐.
  - 초기 유저: 이성 유저 수가 적거나 0이면 후보 0명 → 추천 0건.

---

## 2. 리서치: 빈 상태와 초기 유저 이탈

### 2.1 왜 “에러처럼” 보이면 이탈하는가

- **기대 불일치**: "분석 완료! 최적의 궁합을 찾았습니다" 직후 "불러오지 못했어요" → 성공과 실패가 한 화면에 공존해 혼란과 불신 유발.
- **귀인**: 유저는 "앱이 고장났다" 또는 "내가 뭔가 잘못했다"로 해석할 수 있음.
- **데이팅/매칭 앱**: 첫 인상이 곧 신뢰. "매칭이 안 된다"는 인상은 이탈로 직결.

### 2.2 빈 상태 UX 원칙 (업계 패턴)

- **Empty State ≠ Error State**  
  - 에러: "뭔가 깨졌다" → 재시도/문의 유도.  
  - 빈 상태: "아직 없다" → 기대 설정 + 다음 행동 제시.
- **프레임을 바꾸기 (Reframing)**  
  - "불러오지 못했어요" (실패) → "아직 준비 중이에요" / "곧 채워져요" (진행 중).
- **다음 행동 명확히**  
  - "다시 시도"만 있으면 실패 루프.  
  - "홈/매칭 탭에서 더 보기", "내일 다시 와주세요", "프로필을 완성하면 추천이 빨라져요" 등 **구체적 다음 단계** 제공.
- **초기 유저 특수 처리**  
  - Day-0/1 유저는 추천 0건이 정상일 수 있음.  
  - "곧 새로운 인연이 채워져요", "먼저 홈에서 연애운을 확인해보세요"처럼 **대기 시간을 자연스럽게** 넘기게 함.

### 2.3 목데이터를 쓰지 않는 이유 (노아님 방향)

- 진짜 매칭이 아님 → 기대 이탈과 신뢰 하락.
- 나중에 "목데이터였다"가 드러나면 더 큰 불신.
- **진짜 데이터가 “곧 채워진다”는 스토리 + 명확한 다음 행동**이 초기 이탈 방지에 더 유리.

---

## 3. 제안: 단계별 대응

### 3.1 메시지·프레임 분리 (에러 vs 빈 상태)

| 상황 | 현재 | 제안 (메시지/톤) |
|------|------|-------------------|
| **에러** (EF 실패, 네트워크 등) | "추천 프로필을 불러오지 못했어요" | "잠시 문제가 생겼어요. 아래에서 다시 시도하거나, 매칭 탭으로 가면 곧 확인할 수 있어요." (실패가 아닌 “일시적” 프레임) |
| **빈 리스트** (추천 0건) | "아직 매칭 상대가 없어요 / 조금만 기다려 주세요" | 유지하되, **부제** 추가: "내일 오면 새로운 인연이 채워져요" 또는 "홈에서 오늘의 연애운을 먼저 확인해보세요" |

### 3.2 분석 완료 → 추천 리스트 화면 구조 제안

- **상단 (성공 축하)**  
  - "분석 완료!" / "최적의 궁합을 가진 상대를 찾았습니다"  
  - **보조 문구 한 줄 추가**:  
    - "처음이시라면, 내일부터 추천이 채워져요" (초기 유저)  
    - 또는 "곧 이곳에 운명의 인연이 나타날 거예요"
- **중앙**  
  - **에러 시**: "잠시 문제가 생겼어요" + "다시 시도" + **"매칭 탭에서 확인하기"** (기존 CTA와 동일 흐름).  
  - **빈 리스트 시**: "아직 매칭 상대가 없어요" + "내일 다시 와주시면 새로운 인연이 채워져 있어요" + **동일 CTA**.
- **하단 CTA**  
  - "매칭 탭에서 확인하기" 유지 → 빈 상태여도 **다음 행동이 같아** 흐름이 끊기지 않음.

이렇게 하면 “분석은 성공했고, 추천만 아직 비어 있다”는 인식으로 바뀌어, 에러로 느끼지 않게 할 수 있음.

### 3.3 홈 "받은 좋아요" 빈 상태

- 현재 "아직 없어요"는 적절함.
- 선택: "첫 좋아요를 기다리고 있어요"처럼 **기대감**을 살리는 문구로 소폭 개선 가능.

---

## 4. 기술 측 보완 (원인 제거)

- **`generate-daily-recommendations`** Edge Function 배포  
  - 미배포 시 추천 리스트 로드가 404로 실패함.  
  - 배포 후에는 "에러" 대신 "빈 리스트"가 나오는 경우가 많아짐.
- **배치 궁합 타이밍**  
  - 사주 분석 직후 비동기이므로, 직후 바로 추천 리스트에 반영되지 않을 수 있음.  
  - "곧 채워져요" 메시지가 이 대기 구간을 자연스럽게 커버.

---

## 5. 요약 권장안

1. **에러와 빈 상태 문구 분리**  
   - 에러: "잠시 문제가 생겼어요" + 다시 시도 + "매칭 탭에서 확인하기".  
   - 빈 리스트: "아직 매칭 상대가 없어요" + "내일/곧 채워져요" 부제.
2. **분석 완료 상단에 기대 설정 한 줄**  
   - "처음이시라면 내일부터 추천이 채워져요" 또는 "곧 이곳에 운명의 인연이 나타날 거예요".
3. **목데이터 없이**  
   - "아직 없음"을 정상 시나리오로 프레임하고, 다음 행동(매칭 탭, 연애운 등)을 명확히 제시.
4. **배포**  
   - `generate-daily-recommendations` 배포로 404 제거 → 빈 리스트 플로우로 많이 넘어가게 함.

이대로 적용하면 초반 유저도 "앱이 망가진 것"이 아니라 "아직 채워지는 중"으로 인식해 이탈을 줄이는 방향이 됨.
